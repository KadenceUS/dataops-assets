Project Overview¶

Authors: Brian Prager, Hassen Morad, Bridgette LaFaye
Last Modified: August 20, 2025

Executive Summary¶

Kadence’s current data operations environment is fragmented, with approximately 40 codebases that are similar but not identical. This has led to inconsistent logic, lost bug fixes, hard-coded workarounds, and scattered templates, all of which increase maintenance costs and slow development. The lack of object-oriented design, documentation, and standard operating procedures makes improvements and troubleshooting difficult, particularly under tight deadlines and live deployments.

This initiative will modernize our system by consolidating and restructuring the codebase into modular, testable components. Configuration-driven project creation, standardized templates, and improved documentation will reduce duplication, streamline onboarding, and support faster, more reliable project delivery. Ultimately, these changes will free the data team to focus on higher-value analysis while improving the stability and quality of Kadence’s products.

Project Outline¶

The following goals are presented in order of development priority. Each step includes rationale, a rough estimate on development timelines, and realistic expectations.

1. Converting Code to Object-Oriented Structure¶

Rationale
Object-oriented design (OOD) makes code modular, which brings several key advantages. Modular code is easier to reuse across projects, as developers can combine existing components to meet new requirements instead of regularly reinventing the wheel. It is also easier to test and debug, since errors can be isolated within specific objects rather than traced through a long stream of commands. By enforcing clear rules for how data enters and leaves objects, OOD reduces the risk of sweeping code changes when addressing bugs or edge cases.

OOD also improves collaboration and maintainability. With code organized into clear, well-defined components, developers can work on different parts of a system without interfering with one another. This structure also makes it easier for new team members to understand and contribute, reducing the learning curve and lowering the chance of introducing errors. It also means that creating one clear standard operating procedure (SOP) document much simpler.

Finally, OOD makes it easier to scale solutions and adapt to new data sources or methods. A single-purpose script might need to be replaced entirely when requirements change, but a modular OOD framework allows incremental adjustments with minimal disruption. Together, these benefits improve reliability, adaptability, and long-term efficiency.

Estimated Timeline ~1 Month

Realistic Expectations

  • Requires upfront refactoring, but improves long-term codebase stability.
  • May reveal existing design flaws that must be addressed before progress.
  • Involves a learning curve for those new to object-oriented coding.
  • Development will take time to ensure minimal disruption and maximum usefulness.
  • Carries a significant upfront cost and QA effort, but will save time over the long run.

2. Configuration-Driven Project Creation¶

Rationale
Shifting from hard-coded logic to configuration files allows new projects to be created or updated with minimal developer intervention.

Estimated Timeline ~2 weeks

Realistic Expectations
- Reduces manual coding for project setup.
- Some edge cases will still require custom development.

3. Creating/Updating Templates for Raw Data Schema and Configuration Files¶

Rationale
Standardized templates ensure consistency across projects and reduce onboarding time for new developers.

Estimated Timeline ~1 week

Realistic Expectations
- Templates will evolve as more use cases appear.
- Full coverage of all input variations may not be possible initially.

4. Automating the ETL Pipeline with Hooks for Live Data & Code Updates¶

Rationale
Automation ensures reliability and timeliness of data updates while minimizing manual intervention.

Estimated Timeline ~2 weeks

Realistic Expectations
- Automation will cover the majority of standard workflows.
- Exceptions may require monitoring or manual fixes.
- Hooks must be thoroughly tested to avoid silent failures.

5. Creating Unit Tests¶

Rationale
Unit tests improve reliability, catch regressions, and support continuous integration.

Estimated Timeline ~1 Month

Realistic Expectations
- Full test coverage is unrealistic at the start; focus will be on critical paths.
- Test suite will expand over time as features stabilize.

Persistent Issues¶

Despite the improvements above, some issues will likely persist due to data quality issues, the organizational structure, and lack of a standard software development life cycle (SDLC).

Data Quality Concerns¶

Part of what draws clients to Kadence is the flexibility it shows in setting up new projects and in managing their needs. This presents a significant challenges for the data team, however. For small projects, this can be managed by hotfixes and signficiant human oversight, but presents exponentially more difficulties as the number of projects grow or projects scale up in complexity and size.

A few particular reasons why this flexibility creates difficulty on the backend are:

Schema Drift & Edge Cases¶

Input and output data formats may evolve faster than templates can be updated and properly organized. We aim to move to a configuration driven design, but if each project requires numerous unique templates to function, the likelihood of error increases. We should note that configuration driven approaches are easier to debug, both for a technical and non-technical team member, so we expeect the overall error rates to decrease. The company design choice to allow very flexible products will still pose a challenge going forward as edge cases persist.

Human Data Entry & Format Drift¶

Manual inputs often lead to inconsistencies, such as typos, missing values, or incorrect formats, which in turn break automated processes that expect standardized data. Unique formatting for specific client requests can cause similar issues, as back-end logic may require a programmatic approach to how data is fed into the pipeline.

Over time, even small variations—like different date formats, naming conventions, or field structures—compound into larger integrity issues that are costly to detect and resolve. These errors increase the likelihood of downstream failures in pipelines, make debugging more difficult, and reduce confidence in analytics. Without strong controls or a more generous development lifecycle, documentation and validation can suffer.

We note that the proposed move to Mendix will help reduce the impact of this category of error, as input is managed through a front-end application that can standardize the process of ingesting data.

Organizational Structure¶

The current organizational structure at Kadence for launching projects, providing technical support, and managing ongoing needs differs from more traditional technology companies. While this approach enables speed and flexibility, it also makes it harder to benchmark effectiveness and manage error rates.

Larger data teams often use frameworks like Agile, Kanban, or Scrumban, which provide defined roles and processes to safeguard clients from errors. For example, project managers facilitate communication between technical and domain experts, ensuring feature development aligns with business goals and timelines.

Similarly, dedicated quality assurance professionals focus on identifying flaws and vulnerabilities, allowing developers to concentrate on building functionality. Errors are inherent in coding, and without specialized oversight, even experienced developers can miss issues while managing multiple moving parts.

At Kadence, projects are quickly stood up using code form previous projects and alot of human support in live production environments. This accelerates access to deliverables but bypasses typical IT staging, testing, and validation steps. As a result, the risk of coding errors or misaligned business objectives reaching production is higher, making direct comparisons to traditional IT structures challenging.

Software Development Life Cycle¶

By skipping a formal Software Development Life Cycle (SDLC) and exposing data directly to project managers and clients, Kadence increases the risk of errors and inconsistencies. Without structured stages for requirements, design, testing, and validation, issues like coding bugs, misaligned business logic, or incomplete documentation are more likely. While this approach accelerates delivery, it reduces safeguards, making debugging, maintenance, and reliable feature rollout more challenging.

Live rollout also means that requests from project managers or clients may be incorporated without proper design or scoping. This can lead to overly specific code patches that work for a single feature but introduce unintended issues when edge cases arise. It also complicates code reuse across projects, resulting in highly divergent codebases.

Many of these risks can be mitigated through proper scoping and design during an SDLC. In practice, a balance must be struck between SDLC rigor and rapid delivery. The transition to a new, modular codebase presents an opportunity to streamline SDLC processes, enabling faster development while incorporating the necessary safeguards when lead time allows.
